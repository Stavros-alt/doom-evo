## Phase 2: Sub-par Hardware Optimizations (Conservative)
These are IN-PLACE changes that reduce load but don't change architecture.

### 2.1 Add simple resolution scaling
- In main.py: add `DISPLAY_SCALE = 0.5` constant near top
- Change screen creation to render to a smaller surface (640x480 base) then scale to window size
- Keep window itself resizable; internal render resolution fixed at 640x480 to save GPU
- Comment: "rendering scale for bad computers. change at your own risk"

### 2.2 Reduce particle counts (easiest win)
- In game.py `_spawn_muzzle_particles`: reduce from 2→1
- In `_spawn_hit_particles`: reduce from 5→3
- In `_spawn_death_particles`: reduce from 15→8
- Add comment near each: "cut particles for fps on chromebooks. still looks ok from far"

### 2.3 Cap enemies and bullets (Chromebook melt protection)
- In GameEngine.__init__: cap total enemies to `min(len(enemy_classes), 8)`
- In _update_bullets: if len(bullets) > 25, remove oldest non-player bullet (skip player bullets if you want, or just remove oldest regardless)
- Add comment: "enemy/bullet caps. old code didn't have this but chromebooks die"

### 2.4 Throttle minimap updates (cheap perf)
- In main.py main loop: only call draw_minimap every 0.5 seconds (track last_minimap_update time)
- Add comment: "minimap is expensive. only update sometimes"

### 2.5 Skip fancy wall effects on low quality
- In renderer.py render_frame: add check for a global `LOW_QUALITY` flag (set in main.py)
- If LOW_QUALITY: skip the scanline texture modulation (lines 101-110 in renderer.py) and use plain shaded walls
- Comment: "scanlines are expensive. turn off for bad gpu"

**LOW_QUALITY mode** will be enabled by default in main.py, with a note that users can set it to False if they have better hardware. This looks like a quick hack a student would add last-minute to make it playable on their school Chromebook.
